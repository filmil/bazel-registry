# Instructions

Firstly, read README.md to understand how to build and what the purpose of
things is.

Never auto-modify the following files:
* //tools/bazel
* Any dotfile.
* Any `*.lock` files.
* Any `*.nix` files.

Unless otherwise instructed, only apply maintenance tasks to files in the git
index, or uncommitted files, to avoid redoing work on files that are already
committed to git.

# General change rules

Read the file `//docs/AI_GIT.md` for instructions.

# Documentation

Use Doxygen rules for documenting.

Whenever you add Doxygen documentation, also add the source filegroup targets ,
or source files where filegroups are unavailable, to the `srcs` attribute of
the `doxygen` target named "//:docs", so that doxygen docs could be updated
too. Include all VHDL files, but also C headers, and any other program source
files which contain documentation.

Do not run buildifier, as it will mess up the VHDL file ordering.

When updating documentation run `bazel build //:docs` to verify that it is
correct.

# License maintenance

When maintaining the license files do not modify the following:

* Files matching `*.gtkw`.
* Files under the directory `//third_party`.
* Any files with filenames beginning with a dot.

For all source files and all BUILD files, verify that they have a license
reference at the beginning of the file.

If a file does not have a license reference, add the following text in the
header, appropriately enclosed in comments that are appropriate for the
source file type in question:

```
LICENSE sha256: c71d239df91726fc519c6eb72d318ec65820627232b2f796219e87dcf35d0ab4
```

# `//third_party` maintenance

Every subdir under `//third_party` must have a LICENSE file with the appropriate
license copied from its source distribution.

# Public API documentation maintenance

Ensure that the repository is clean before starting this procedure.

For all source files, we want to maintain an up-to-date documentation of their
respective public API.

# Publish to Bazel central registry

Read the file `//docs/AI_BCR.md` for publication instructions.



# Fragment for handling go code in bazel

## General

- Use bazel_dep named `rules_go`. Do not rename it. Use at least version
  `0.60.0`.
- Use at least go SDK version `1.26.2`.

## Substitutes

- Use `bazel run @rules_go//go -- ARGS` instead of `go ARGS` in all cases.
  Example: `bazel run @rules_go//go -- mod tidy`.


##

- Configure the workspace for `rules_go`, `gazelle`, `rules_proto`, `protobuf`, and `grpc`.
- Enforce canonical repo names: `@rules_go//go` and `@grpc`.
- Configure `protoc` as a prebuilt Bazel toolchain.
- Use `bazel mod tidy` to update module versions in `MODULE.bazel`.
- Use `bazel run //:gazelle` to generate and update build rules.
- Set up a `.bazelrc` that loads `user.bazelrc` if it exists.



# Fragment for setting up a standardized git workflow

## General Change Rules

- Do not change files in `//third_party` without explicit permission from the user.


## Commits

* Use "conventional commits v1.0.0" for the commit title.
- Append the following note to the last line of the commit message (below any
  summaries):
  ```
  This commit has been created by an automated coding
  assistant, with human supervision.
  ```
- Additionally, append the exact prompt used to generate the commit in full.


## Rebase and Merges

- Prefer rebasing the current branch on top of another instead of merging into
  a branch.
- E.g., to get content from `main` from the `origin` repo, use `git rebase
  --pull origin main`.
- **CRITICAL**: Always use the `--no-edit` option when running `git merge`,
  `git cherry-pick`, or `git rebase` (e.g., `git merge --no-edit`, `git rebase
  --no-edit`) to prevent interactive editors from blocking automated tasks.


## Pull Requests

- Use the `gh` utility to create the pull request.
- Use `origin/main` as a baseline for the pull request.
- Rebase the local `main` branch from `origin/main`.
- Create a new branch from `main` using the pattern
  `ai-dev-YYYYMMDD-NONCE-SHORT_TOPIC`, where `NONCE` is replaced with a random
  sequence of letters generated by executing the bundled script at
  `scripts/generate_nonce.sh`, and `SHORT_TOPIC` is replaced with an extremely
  short summary of the change.
- Push this new branch to the remote.
- Create the pull request using `gh`. The PR description must contain a summary
  of all commits between `origin/main` and the top of the branch.
- The PR description must also contain the following note at the end:
  ```
  This pull request has been created by an automated coding assistant, with
  human supervision.
  ```
- Additionally, append the exact prompt used to generate the pull request in
  full.
- The task is complete once the `gh` command finishes successfully.
- When fixing a specific Github issue, put a line "Fixes: URL", where URL
  is the link to the Github isssue being fixed.
